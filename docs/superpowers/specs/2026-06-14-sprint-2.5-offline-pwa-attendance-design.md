# Sprint 2.5 — Offline PWA Attendance (Design)

- **Date:** 2026-06-14
- **Status:** Approved (brainstorming complete) — ready for implementation plan
- **Builds on:** Sprint 2 Attendance (merged to `main`, browser-QA'd). See `docs/RESUME.md`.

## Goal

Let a teacher mark attendance in a connectivity dead zone and have nothing lost.
Marks are durable the instant they're tapped and sync automatically when the
device regains a connection. The realistic flow: a teacher loads their classes
in the morning on office wifi, then marks throughout the day in areas with no
signal; everything reconciles when they're back in coverage.

## Scope (locked decisions)

1. **Coverage: write + same-day read.** Mark offline, and re-open *today's* grid
   offline for any class loaded while online (roster cached on load). Out of
   scope: cold-loading a brand-new class's roster while fully offline.
2. **Write path: offline-first.** Every tap writes to a local IndexedDB queue
   first, then a syncer flushes to the API. One code path online and offline.
3. **Conflict handling: replay, last-write-wins by sync order.** Queued marks
   replay to the existing endpoint; whoever syncs last wins. **Zero backend
   change.** Replay-safety comes from the server's `(studentId, date)` upsert
   being inherently idempotent.
4. **Mechanism: app-layer IndexedDB queue + reconnect syncer (Approach A).** No
   Service Worker Background Sync (unsupported on iOS Safari, harder to test).
   The existing `public/sw.js` (shell caching) is untouched. Background Sync is
   a documented future enhancement, not part of 2.5.

### Non-goals
- Background sync while the app/tab is fully closed.
- Cross-device conflict detection or resolution UI.
- Caching all classes for cold fully-offline loading.
- Any change to the attendance API or database schema.

## Architecture

A self-contained offline layer under `apps/web/src/lib/offline/`. The marking
grid (`apps/web/src/app/(app)/attendance/page.tsx`) is its only consumer. The
NestJS API and `public/sw.js` are not modified.

### Modules

- **`offline/db.ts`** — opens one IndexedDB database via `idb`, with three
  object stores:
  - `mark_queue` — key: `"{classId}|{date}|{studentId}"`.
    Value: `{ classId, date, studentId, status, idempotencyKey, queuedAt }`.
    The composite key gives free coalescing: re-tapping a student before a sync
    overwrites the queued entry, so only the final status is ever sent.
  - `roster_cache` — key: `"{classId}|{date}"`. Value: `AttendanceDay` + `cachedAt`.
    Written on every successful online roster load; read on network failure.
  - `classes_cache` — key: `"classes"`. Value: `Class[]`. Keeps the class
    `Select` populated offline.
- **`offline/queue.ts`** — `enqueue(mark)`, `getAll()`, `remove(keys)`.
  Generates `idempotencyKey` (`crypto.randomUUID()`) at enqueue.
- **`offline/roster-cache.ts`** — `cacheRoster/getRoster`, `cacheClasses/getClasses`.
- **`offline/syncer.ts`** — a singleton.
  - `flush()`: read queue → group pending marks by `(classId, date)` → one
    `api.markAttendance` batch per group → on 2xx delete those keys; on failure
    keep them and schedule a backoff retry (2s → 4s → 8s, capped).
  - Triggers: app load, the `online` event, and after each `enqueue` when
    `navigator.onLine`.
  - Emits `{ online, pendingCount, state }` to subscribers
    (`state ∈ idle | syncing | offline | error`).
- **`offline/useOfflineSync.ts`** — React hook exposing the syncer's state to the
  grid.

### Type change
Add optional `idempotencyKey?: string` to `MarkAttendanceRecord` in
`apps/web/src/lib/api.ts`. The API DTO already accepts it. No other API change.

## Data flow

### Marking a student
1. Tap → optimistic local state update (instant UI).
2. `enqueue()` writes to `mark_queue` (durable immediately — survives tab close,
   reload, device death).
3. If `navigator.onLine`, fire `syncer.flush()` immediately. This replaces the
   current 600 ms debounce; coalescing now happens in the queue, not via a timer.

### Syncing
- `flush()` groups pending marks by `(classId, date)`, sends one
  `api.markAttendance` batch per group, deletes synced keys on success, retries
  failed groups whole with backoff and on next reconnect.

### Loading the grid
- **Classes:** `api.listClasses()` → cache on success; read cache on failure.
- **Roster:** `api.getClassAttendance()` → cache on success; read
  `roster_cache[(classId,date)]` on failure.
- **Overlay:** after loading the server/cached roster, overlay any pending queued
  marks for that `(classId, date)` on top, so the grid always shows the teacher's
  latest local taps even before they sync. The queue is the source of truth for
  unsynced edits.

## Error & edge handling

- **Sync network failure** → marks stay queued; UI shows "waiting to sync";
  auto-retry with backoff + on reconnect.
- **Token expiry mid-offline** (rare; 30-day JWT) → existing `authedRequest` 401
  redirect to `/login` stands; the queue persists in IndexedDB, so the app-load
  flush drains it after re-login. Nothing lost.
- **Roster cache miss offline** (class never loaded online) → existing "Could not
  load" error state with retry. This is the deliberate boundary of same-day read.
- **Failed/incomplete batch** → a failed group is retried whole. Safe even if
  some rows in the group already committed before the failure, because each
  `(studentId, date)` upsert is idempotent — re-sending an already-saved row
  just re-sets the same value (a no-op).

## UI states

Offline is normal in the target market — the UI is calm, not alarming, and built
from the existing design system (Bold Ink + Saffron, `Badge`, motion tokens,
`aria-live="polite"`).

- **Connection indicator:** a quiet "Offline" pill near the grid header when
  `!navigator.onLine`; absent when online.
- **Sync indicator** (replaces the current `saveState` span), driven by
  `useOfflineSync()`:
  - online, nothing pending → subtle "All saved", fades.
  - syncing → "Syncing N…".
  - offline with pending → "N saved on this device" (reassurance, not error).
  - sync failed → "Sync failed — retrying" (warning tone, self-healing).

## Testing

Closes the `apps/web` test-framework gap (in-scope: the queue/syncer logic must
be proven). UI tests currently live only in `packages/ui` (vitest); API uses jest.

- **Bootstrap:** vitest + @testing-library/react + fake-indexeddb in `apps/web`;
  add a `test` script so Turborepo's `turbo run test` (and CI) picks it up.
- **Unit tests:**
  - `queue`: enqueue, coalesce (re-tap overwrites, not appends), remove, getAll.
  - `roster-cache`: cache/get roster and classes.
  - `syncer`: online flush clears queue; offline → no-op, stays queued; reconnect
    triggers flush; batches grouped by `(classId, date)`; partial failure keeps
    group + backoff; idempotent replay.
  - overlay: queued marks render on top of a stale/cached roster.
- **Offline browser QA** (third safety net, same as Sprint 2): load grid online →
  emulate offline (CDP/Playwright) → mark → confirm queued + reassuring UI +
  roster persists across an offline reload → go online → confirm one batch POST
  per `(class,date)` hits the API and the indicator settles to "All saved."

## Out-of-scope future enhancements
- Service Worker Background Sync for sync-while-closed on supporting browsers.
- Caching all rosters for cold fully-offline class loading.
- Capture-time last-write-wins or conflict resolution UI.
